Моим клинт-серверным приложением будет веб-сайт одного человека. Что-то вроде его блога,
где он может добавлять посты, фотографии, опросы и подобные штуки. Любой пользователь может
зайти, зарегистрироваться/залогиниться и оставлять комментарии под контентом. Необходимо
реализовать взаимодействие системы с пользователем, то есть его кабинет, вход, так же исполнение 
процедур программных скриптов, которые поддерживают построение приложения и взаимодействие с базой 
данных(далее я использую сокращение БД:), в которой хранится все информация: посты, пользователи, комментарии. 

Архитектура будет трехуровневая. То есть:
Клиент будет отвечать за интерфейс, авторизацию и графическое отбражение запросов(например посты
за определенный преиод, комментарии отдельного пользователя)

Сервер приложений будет связующим звеном, отвечает за программную часть сайта. То есть вся логика 
связей страниц будет зафиксированна именно здесь. 

Сервер БД будет получать запросы на чтение или запись данных в БД, реализовывать их, обращаясь к БД
напрямую и получать результаты.

Все компоненты будут связаны между собой, то есть клиент с сервером приложений, сервер приложений 
с сервером БД и сервер БД с БД

Иллюстративый пример:

Пользователь логинится. На клиенте его логин и пароль принимаются, пароль шифруется и передаются 
на сервер приложений, оттуда логин отправляется на сервер БД, где по логину мы запрашиваем пароль, 
занесенный ранее в БД. Сервер БД отправляет эту информацию на сервер приложений, где эти 2 пароля
сравниваются. Затем ответ о совпадении или нет отправляется в интерфейс пользователя.

Пользователь оставляет комментарий, нажимает кнопку "опубликовать". Сервер приложений принимает всю
информацию,то есть, кто оставил комментарий, когда, под каким постом и формирует запрос на запись
данных. Сервер сохраняет записи (или нет) и как результат получаем сообщение об успехе (или нет) на 
сервер приложений, откуда согласно програмному коду пользователь получает сообщение на клиенте.

Такая система удобна в данном случае, так как у каждого элемента своя зона ответственности.Это 
позволяет ставить проверки безопасности на каждом уровне, разрабатывать их отдельно друг от друга,
масштабировать будет не так сложно. 
У системы есть нюансы, вроде отказа сервера, но их можно избезать с помощью кэширования. 

Клиент не имеет прямого доступа к базе данных, что позволяет делать их взаимодействие более безопасным.
Сервер базы данных при наличии необходимого программного кода безопасности может обеспечивать защиту от 
ошибок: 
грязного чтения
невоиспроизводимого чтения 
чтения фонтомов
